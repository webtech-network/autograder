# Input/Output Template

**Registry key:** `input_output` · **Sandbox:** Required · **Languages:** Python, Java, Node.js, C++

The input/output template tests console programs that read from stdin and write to stdout. It supports output matching, file artifact verification, crash detection, and forbidden import checking.

## Available Tests

| Test Name | Purpose | Sandbox |
|-----------|---------|---------|
| [`expect_output`](#expect_output) | Check program output against expected value | Yes |
| [`dont_fail`](#dont_fail) | Verify program doesn't crash on given input | Yes |
| [`expect_file_artifact`](#expect_file_artifact) | Check files generated by the program | Yes |
| [`forbidden_import`](#forbidden_import) | Detect use of forbidden libraries | No |

---

## expect_output

Runs the student's program with given inputs and compares the output to an expected value. Both actual and expected outputs are trimmed of leading/trailing whitespace before comparison.

### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `inputs` | list of strings | Yes | — | Lines fed to stdin, one per entry |
| `expected_output` | string | Yes | `""` | Expected stdout output |
| `program_command` | string | No | Auto-resolved | Command to execute the program |

### Scoring

- **100** if `actual_output.strip() == expected_output.strip()`
- **0** if output doesn't match, or on timeout, compilation error, runtime error, or system error

### Command Resolution

If `program_command` is not provided, the autograder resolves it automatically based on the submission language:

| Language | Resolved Command |
|----------|-----------------|
| Python | `python3 {file}` |
| Java | `javac {file} && java {class}` |
| Node.js | `node {file}` |
| C++ | `g++ -o program {file} && ./program` |

You can also provide a language-specific command map:

```json
{
  "name": "program_command",
  "value": {
    "python": "python3 solution.py",
    "java": "javac Solution.java && java Solution",
    "node": "node solution.js",
    "cpp": "g++ -o solution solution.cpp && ./solution"
  }
}
```

### Example

```json
{
  "name": "expect_output",
  "parameters": [
    { "name": "inputs", "value": ["5", "3"] },
    { "name": "expected_output", "value": "8" },
    { "name": "program_command", "value": "python3 calculator.py" }
  ],
  "weight": 50
}
```

!!! tip "Multiple inputs"
    Each entry in the `inputs` list is sent as a separate line to stdin. For a program that reads two numbers, use `["5", "3"]`.

## dont_fail

Runs the student's program and checks that it completes without crashing. The actual output is ignored — only the exit status matters.

### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `user_input` | string | No | `""` | Single input string fed to stdin |
| `program_command` | string | No | Auto-resolved | Command to execute the program |

### Scoring

- **100** if the program completes without timeout, compilation error, runtime error, or system error
- **0** on any error

### Example

```json
{
  "name": "dont_fail",
  "parameters": [
    { "name": "user_input", "value": "-1" },
    { "name": "program_command", "value": "python3 factorial.py" }
  ]
}
```

!!! note "Use case"
    `dont_fail` is ideal for testing edge cases where you want to ensure the program handles unusual input gracefully, without checking the exact output.

## expect_file_artifact

Runs the student's program and then checks a file it generated inside the sandbox. Supports exact matching, substring matching, and regex matching.

### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `artifact_path` | string | Yes | `""` | Relative path to the generated file inside the sandbox |
| `expected_content` | string | Yes | `""` | Expected file content |
| `match_mode` | string | No | `"exact"` | How to compare: `"exact"`, `"contains"`, or `"regex"` |
| `inputs` | list of strings | No | `None` | Lines fed to stdin |
| `program_command` | string | No | Auto-resolved | Command to execute the program |
| `normalization` | boolean | No | `true` | Normalize whitespace and line endings before comparison |

### Match Modes

| Mode | Behavior |
|------|----------|
| `exact` | Full content must match (after optional normalization) |
| `contains` | Expected content must appear as a substring |
| `regex` | Expected content is treated as a regex pattern |

### Normalization

When `normalization` is `true` (default), both actual and expected content are normalized:

- `\r\n` and `\r` are replaced with `\n`
- Trailing whitespace is stripped from each line
- Leading and trailing whitespace is stripped from the full content

### Scoring

- **100** if the file content matches per the selected mode
- **0** on mismatch, file not found, invalid path, invalid regex, execution error, or if the file exceeds 1 MB

### Security

- `artifact_path` must be **relative** (no leading `/`)
- Path traversal (`..`) is rejected
- Maximum file size: **1 MB**

### Example

```json
{
  "name": "expect_file_artifact",
  "parameters": [
    { "name": "program_command", "value": "python3 generate_report.py" },
    { "name": "artifact_path", "value": "output/report.txt" },
    { "name": "expected_content", "value": "Total: 42" },
    { "name": "match_mode", "value": "contains" }
  ]
}
```

## forbidden_import

Scans the student's source code for forbidden library imports. This test does **not** execute the program — it performs static analysis only.

### Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `forbidden_imports` | list of strings | Yes | — | Library/module names to forbid |
| `submission_language` | string | No | Auto-detected | Language of the submission |

### Supported Languages and Detected Patterns

| Language | Import Patterns Detected |
|----------|------------------------|
| Python | `import lib`, `from lib import ...` |
| Java | `import pkg.Class;`, `import static pkg.Class.method;` |
| Node.js | `require('lib')`, `import ... from 'lib'` |
| C/C++ | `#include <lib>`, `#include "lib..."` |

### Scoring

- **100** if no forbidden imports are found (or if the list is empty)
- **0** if any forbidden import is detected

### Example

```json
{
  "name": "forbidden_import",
  "parameters": [
    { "name": "forbidden_imports", "value": ["os", "subprocess", "shutil"] }
  ]
}
```

!!! warning "Penalty test"
    `forbidden_import` is typically used in a **penalty** category. When placed in penalty, students lose points only if they use forbidden libraries.

            "weight": 25
          },
          {

              { "name": "inputs", "value": ["subtract", "10", "4"] },

          {

            ],
            "weight": 25
          },
          {
            "name": "expect_output",
